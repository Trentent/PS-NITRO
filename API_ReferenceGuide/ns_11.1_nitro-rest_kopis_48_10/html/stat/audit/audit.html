auditmemallocfail<Double>

Failures in allocation of Access Gateway context structure. When an Access Gateway session is established, the NetScaler creates an internal context structure , which identifies the user and the IP address from which the user has logged in.

auditmemallocfailrate<Double>

Rate (/s) counter for auditmemallocfail

auditportallocfail<Double>

Number of times the NetScaler failed to allocate a port when sending a syslog message to the syslog server(s).

auditportallocfailrate<Double>

Rate (/s) counter for auditportallocfail

auditcontextnotfound<Double>

Failures in finding the context structure for an Access Gateway session during attempts to send session-specific audit messages. During an Access Gateway session, audit messages related to the session are queued up in the auditlog buffer for transmission to the audit log server(s). If the session is killed before the messages are sent, the context structure allocated at session creation is removed. This structure is needed for sending the queued auditlog messages. If it is not found, this counter is incremented.

auditcontextnotfoundrate<Double>

Rate (/s) counter for auditcontextnotfound

nsbchainallocfail<Double>

Nsb Chain allocation failed.

nsbchainallocfailrate<Double>

Rate (/s) counter for nsbchainallocfail

clientconnfail<Double>

Failures in establishment of a connection between the NetScaler and the auditserver tool (the Netscaler's custom logging tool).

clientconnfailrate<Double>

Rate (/s) counter for clientconnfail

flushcmdcnt<Double>

Auditlog buffer flushes. In a multiprocessor NetScaler, both the main processor and the co-processor can generate auditlog messages and fill up the auditlog buffers. But only the primary processor can free up the buffers by sending auditlog messages to the auditlog server(s). The number of auditlog buffers is fixed. If the co-processor detects that all the auditlog buffers are full, it issues a flush command to the main processor.

flushcmdcntrate<Double>

Rate (/s) counter for flushcmdcnt

systcpconnfail<Double>

Failures in establishment of a connection between the NetScaler and the syslog server.

systcpconnfailrate<Double>

Rate (/s) counter for systcpconnfail

logunsentlbsys<Double>

Total auditlog messages which are not delivered to load balanced syslog servers

logunsentlbsysrate<Double>

Rate (/s) counter for logunsentlbsys

logsdropped<Double>

Operations

You can perform the following operations:

get (all)

Some options that you can use for each operation:

  • Getting warnings in response: NITRO allows you to get warnings in an operation by specifying the "warning" query parameter as "yes". For example, to get warnings while connecting to the NetScaler appliance, the URL is as follows:
    http://<netscaler-ip-address>/nitro/v1/config/lbvserver?warning=yes

    If any, the warnings are displayed in the "X-NITRO-WARNING" header and the HTTP status code is 209.

  • Fetch statistics of the bound resources: NITRO allows you to fetch statistics of bound resources by specifying the "statbindings" query parameter as "yes". This query parameter is applicable only to get(byname) operation. For example, to get binding statistics of a given lbvserver, the URL is as follows:
    http://<netscaler-ip-address>/nitro/v1/stat/lbvserver/lbvip1?statbindings=yes

  • Authenticated access for individual NITRO operations: One of the authentication mechanisms is OneTimelogin/Basic authentication. You can use the below two headers specifying username and password and send it along with the regular nitro request.

    X-NITRO-USER:<username>
    X-NITRO-PASS:<password>

    Note: In such cases, make sure that the request header DOES not include the following:
    Cookie:NITRO_AUTH_TOKEN=<tokenvalue>

NOTE: Mandatory parameters are marked red and bold. Placeholder text is marked in green.

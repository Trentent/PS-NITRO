dnsaction

Configuration for DNS action resource.

Read/write properties

actionname<String>

Name of the dns action.

actiontype<String>

The type of DNS action that is being configured.
Possible values = ViewName, GslbPrefLoc, noop, Drop, Cache_Bypass, Rewrite_Response

ipaddress<String[]>

List of IP address to be returned in case of rewrite_response actiontype. They can be of IPV4 or IPV6 type.
In case of set command We will remove all the IP address previously present in the action and will add new once given in set dns action command.

ttl<Double>

Time to live, in seconds.
Default value: 3600
Minimum value = 0
Maximum value = 2147483647

viewname<String>

The view name that must be used for the given action.

preferredloclist<String[]>

The location list in priority order used for the given action.
Minimum length = 1

dnsprofilename<String>

Name of the DNS profile to be associated with the transaction for which the action is chosen.
Minimum length = 1
Maximum length = 127

Read only properties

drop<String>

The dns packet must be dropped.
Possible values = YES, NO

cachebypass<String>

By pass dns cache for this.
Possible values = YES, NO

builtin<String[]>

Flag to determine whether DNS action is default or not.
Possible values = MODIFIABLE, DELETABLE, IMMUTABLE, PARTITION_ALL

__count<Double>

count parameter

Operations

You can perform the following operations:

adddeleteupdateunsetget (all)getcount

Some options that you can use for each operation:

  • Getting warnings in response: NITRO allows you to get warnings in an operation by specifying the "warning" query parameter as "yes". For example, to get warnings while connecting to the NetScaler appliance, the URL is as follows:
    http://<netscaler-ip-address>/nitro/v1/config/lbvserver?warning=yes

    If any, the warnings are displayed in the "X-NITRO-WARNING" header and the HTTP status code is 209.

  • Authenticated access for individual NITRO operations: One of the authentication mechanisms is OneTimelogin/Basic authentication. You can use the below two headers specifying username and password and send it along with the regular nitro request.

    X-NITRO-USER:<username>
    X-NITRO-PASS:<password>

    Note: In such cases, make sure that the request header DOES not include the following:
    Cookie:NITRO_AUTH_TOKEN=<tokenvalue>
